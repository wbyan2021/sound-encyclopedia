# 声音大百科 · 产品战略文档

> 面向 1-6 岁儿童的开源声音认知产品。通过真实声音帮助孩子认识自然与世界。
> 本文档定义产品定位、现状评估、改进路线和战略决策。
> 技术实现详见 [开发文档](开发文档.md)。

---

## 一、产品定位

**核心承诺**：真实声音 + 极简交互 + 零识字门槛，让孩子在家通过真实声音认识自然与世界。

### 设计原则

- 数据与代码分离：加声音零改代码
- 贡献门槛最低化：从不会 Git 的家长到开发者都能参与
- 零成本部署：GitHub Pages，无后端
- 版权干净：CC0（素材）+ MIT（代码）

### 社区共建闭环

```
社区贡献者 → 提 PR（上传 MP3 + meta.json）
  → GitHub Actions 自动校验
  → 人工 Review → 合并到 main
  → build-manifest.js 自动生成索引
  → 前端 fetch manifest.json 动态渲染
  → 孩子点击 → 播放声音
```

## 二、产品现状评估

> 2026年6月评估。从六个维度审视产品成熟度。

![雷达图](雷达图.svg)

### 2.1 产品价值（8/10）—— 项目最扎实的地方

1-6 岁儿童的声音认知，市面产品基本两类：电子合成音效（假得离谱）或识字 App 顺带放声音（本末倒置）。本产品填补了「真实声音 + 极简交互 + 零识字门槛」的空白。

家长痛点：孩子天天看屏幕，但屏幕给的是视觉刺激，声音这种「被动输入、激发想象」的维度被严重低估。

**扣分点**：价值成立但深度不够。孩子点 38 个动物听一遍，新鲜感过了就没有「为什么要再打开一次」的理由。

### 2.2 内容深度（4/10）—— 最大短板

38 种动物，96 个音频。数量不差，但产品视角看：

- **覆盖窄**：预留了自然/交通/生活 4 大分类，但只有动物有数据。3 个空分类是坏体验。
- **重复访问价值低**：点完一圈就完了。没有「今日推荐」「新声音上线」「按场景组合」这类持续内容。
- **缺少教育纵深**：孩子听到狗叫，然后呢？没有「这是什么狗」「狗为什么这么叫」这种从声音延伸的知识。声音是钩子，知识才是留住用户的鱼线。

**方向**：别急着扩分类，先把动物这条线做深。每个动物加一段 30-50 字儿童友好科普文案（`fun_fact` 字段），点击后能展开看。这比再加 20 种动物的 ROI 高得多。

### 2.3 用户体验（7/10）—— 形态对了，细节还差口气

**做对的**：
- 开始页 → 加载 → 主界面流程清晰
- 点击即播放，符合儿童使用习惯
- 动画反馈（弹跳+星星+波纹）给得很足
- 音量控制是刚需

**差口气的地方**（2026-06-29 审计更新）：

| 问题 | 严重度 | 状态 |
|------|--------|------|
| ~~预加载 96 个文件~~ | ~~高~~ | ✅ 已改。音频点播加载，数据全量 fetch manifest 但体量尚可 |
| ~~没有播放状态指示~~ | ~~中~~ | ✅ 已做。`.playing`/`.played` 三种状态，波纹动画 + emoji 弹跳 |
| ~~无搜索/收藏~~ | ~~中~~ | ✅ 已做。搜索框 + 收藏系统 + localStorage 持久化 |
| ~~SW 缓存策略不当~~ | ~~低~~ | ✅ 已改。manifest: network-first，mp3: cache-first |
| 无家长区 | 中 | ❌ 仍然缺失。家长需要一个入口看「这 App 讲什么/怎么用/贡献声音」 |

### 2.4 技术架构（9/10）—— 隐藏王牌

数据驱动 + 自动校验 + 自动构建 + GitHub Actions + 单仓库零依赖。这个架构的扩展性足够撑到 500+ 声音不用重构。

**扣 1 分**：缺少测试。build-manifest.js 和 validate.js 是整个系统的地基，没有测试用例。哪天脚本崩了，PR 全卡住，排查会很痛。

### 2.5 社区生态（3/10）—— 现在是单机游戏

搭了社区贡献的管道（Issue 模板、PR 模板、Actions、CONTRIBUTING），但管道里没有水。

- 0 个外部贡献者
- 0 个 Issue
- 0 个 PR
- README 写得很好但没人看

开源项目的冷启动比产品本身难 10 倍。现在状态是「精心装修但没挂牌的房子」。

### 2.7 AI 赋能（6/10）—— 基础设施完成，TTS 待全量

2026-06-30 更新：AI 基础设施全部打通，前端已集成 TTS 朗读按钮。

- ✅ 58 条声音的 `fun_fact` 已全部由 AI 生成（`fun_fact_ai_generated: true`），质量达标
- ✅ AI 科普文案批量生成脚本（ai-generate-fun-fact.js）稳定可用
- ✅ AI TTS 朗读生成脚本（ai-generate-tts.js）已实现
- ✅ 本地管理后台 Web UI（admin-server.js）上线，支持可视化操作 AI 生成
- ✅ 前端 TTS 朗读按钮集成完成（中文名/英文名/科普讲解三个独立按钮）
- ✅ 多音频随机轮播前端已实现
- 🔄 TTS 朗读音频全量生成中（当前 10/58，部分声音已有生成音频）
- ⬜ 全量 TTS 生成完成后，AI 赋能将达到 8/10

本轮 AI 赋能聚焦三大核心能力：

1. **AI 内容生成**：根据动物标签（物种、习性、栖息地等）自动调用大模型，生成准确、生动、适合儿童阅读的科普文案（`fun_fact`）与描述内容。批量生成后由人工审核入库。
2. **音频与文字联动**：每段 AI 生成的科普文案对应一条专属 TTS 朗读音频；动物中文名、英文名均可独立触发语音播报，发音清晰自然，帮助孩子独立使用。
3. **多音频随机轮播**：拥有多条真实音频的动物条目，每次点击或页面切换时从可用音频中随机抽取一条播放，避免重复，提升探索乐趣。

以上三项是 AI 功能落地的**最低可用集（MVP）**，优先于猜声音游戏、声音故事等进阶玩法。

AI 不做的事：生成动物叫声等合成声音（违背「真实声音」原则）、替代人类审核（版权复核必须人工）。AI 朗读仅用于文字内容，且需明确标注「🤖 AI 朗读」。

### 2.6 可持续性（5/10）—— 最该想清楚的问题

一个残酷的事实：纯开源 + 纯免费 + 无商业模式 = 大概率死掉。不是诅咒，是开源项目的统计规律。

**必须回答的问题**：三年后，谁在维护这个项目？动力是什么？

可能的答案（不冲突）：
- **公益基金/教育机构赞助**：定位成「儿童数字启蒙公共资源」，找教育基金会谈
- **增值服务**：声音库永久免费，但「无网络离线包」「亲子课程包」收费
- **企业 CSR**：找字节/腾讯的教育公益部门，让他们把维护接过去
- **纯个人热爱**：自己维护，但要有心理准备

**建议**：先别想商业模式，但**现在就写「项目可持续性声明」**放进 README。说清楚：这是公益项目、维护者是谁、如何贡献、如果维护者不干了怎么办。透明比承诺重要。

### 4.2 Phase B：AI 内容赋能（详细实现）

#### 4.2.1 数据模型扩展（与现有 meta.json 兼容）

每个声音的 `meta.json` 已支持 `fun_fact` 和 `fun_fact_ai_generated`。本阶段扩展如下：

```json
{
  "id": "animals.dog",
  "name_zh": "狗",
  "name_en": "Dog",
  "fun_fact": "小狗的鼻子比人厉害一万倍，你藏在口袋里的小饼干它一下就能闻到哦",
  "fun_fact_ai_generated": true,
  "ai": {
    "generated_at": "2026-06-29T12:00:00Z",
    "model": "MiniMax-M3",
    "prompt_version": "v2",
    "text_hash": "sha256:abc123..."
  },
  "tts": {
    "name_zh": "generated/name-zh.mp3",
    "name_en": "generated/name-en.mp3",
    "fun_fact": "generated/fun-fact.mp3",
    "voice_id_zh": "female-tianmei",
    "voice_id_en": "male-qn-qingse",
    "generated_at": "2026-06-29T12:00:00Z"
  }
}
```

新增字段说明：
- `fun_fact_ai_generated`：标识是否 AI 生成，前端展示「🤖」角标。
- `ai.generated_at`：生成时间，用于判断是否需要重新生成。
- `ai.model`：记录使用的模型，便于追溯与成本核算。
- `ai.prompt_version`：prompt 版本，便于后续批量优化。
- `ai.text_hash`：文本哈希，用于避免内容无变化时重复调用 API。
- `tts.*`：AI 朗读音频路径与生成信息；真实动物声音仍放在 `audio/` 目录下，AI 朗读音频统一放在 `generated/` 目录下，避免混淆。

#### 4.2.2 AI 科普文案与描述生成

调用 MiniMax LLM（`/v1/chat/completions`，OpenAI 兼容格式）。输入以动物标签为核心（物种、习性、栖息地、分类、子分类），让生成内容更准确、有针对性。

**Prompt 模板（v1）：**

```text
你是一位儿童科普作家，擅长为 1-6 岁儿童撰写简短、有趣、准确的科普小知识。
请根据以下动物标签，写一段 30-50 字的中文科普文案。

要求：
- 只陈述科学事实，不编造；
- 语言口语化、有画面感，孩子能听懂；
- 充分利用提供的标签（物种、习性、栖息地等），突出一个有趣的冷知识；
- 不输出任何解释、标题或多余内容，只返回一段文案。

输入：
- 中文名：{name_zh}
- 英文名：{name_en}
- 分类：{category} / {subcategory}
- 标签：{tags}
- 栖息地：{habitat}
- 现有描述：{description}
```

**描述生成 Prompt（可选）：**

```text
请根据以下标签，为声音条目写一段不超过 30 字的简短描述，让孩子和家长一眼就知道这是什么声音。
输入：中文名、英文名、分类、标签。
```

**触发方式：**

| 方式 | 场景 | 实现位置 |
|------|------|---------|
| 本地脚本批量生成 | 维护者本地为全部/指定声音生成文案 | `scripts/ai-generate-fun-fact.js` |
| GitHub Actions 自动补全 | PR 合并后扫描缺失 `fun_fact` 的条目 | `.github/workflows/ai-generate.yml` |
| 管理后台单条生成 | 本地开发时为某个声音重新生成 | 可选本地管理工具 |

**审核流程：**

```
AI 生成文案 → 写入临时分支 → 人工 review 修改 → 合并到 main → build-manifest 重新生成 manifest
```

#### 4.2.3 AI 文字朗读（TTS）

为已生成的文字内容生成朗读音频，使用 MiniMax TTS API（`/v1/t2a_v2`）。中文名与英文名各自独立生成、独立触发，方便孩子学习双语发音。

**朗读内容优先级：**

| 内容 | 触发场景 | 文件名约定 | 是否缓存 |
|------|---------|-----------|---------|
| 中文名 `name_zh` | 详情页「🔊 读中文名」按钮，独立触发 | `{sound_id}/generated/name-zh.mp3` | 是 |
| 英文名 `name_en` | 详情页「🔊 读英文名」按钮，独立触发 | `{sound_id}/generated/name-en.mp3` | 是 |
| 科普文案 `fun_fact` | 详情页「🔊 听 AI 讲解」按钮 | `{sound_id}/generated/fun-fact.mp3` | 是 |
| 描述 `description` | 详情页「🔊 读描述」按钮（可选） | `{sound_id}/generated/description.mp3` | 是 |
| 场景介绍 | 场景播放页 | `data/scenes/{scene_id}/generated/intro.mp3` | 是 |
| 故事文字 | 故事模式 | `data/stories/{story_id}/generated/story.mp3` | 是 |

**前端交互要求：**
- 真实动物声音与 AI 朗读在 UI 上明确区分。
- 中文名、英文名、科普文案三个朗读按钮独立存在，可单独触发。
- TTS 音频默认不预加载，点击后再请求，避免拖慢首屏。

**技术要点：**

- 音色选择：中文使用温柔女声（如 `female-tianmei`），英文使用清晰发音（如 `male-qn-qingse`），儿童友好、语速偏慢。
- 语速控制在 0.9~1.0，避免过快。
- 生成音频保存到 `data/sounds/{category}/{id}/generated/` 或 `data/scenes/{id}/generated/`、`data/stories/{id}/generated/`，随仓库一起提交。
- 前端默认播放真实动物声音；朗读按钮独立触发，UI 上明确区分「真实声音」与「AI 朗读」。

**新增脚本：**

- `scripts/ai-generate-tts.js`：批量为已有 `fun_fact` 和名称生成 TTS 音频。
- `scripts/ai-generate-fun-fact.js` 生成文案后，可联动调用 TTS 生成。

#### 4.2.4 AI 场景组合

目标：解决「重复访问价值低、缺少持续内容」的问题。

新增 `data/scenes.json`：

```json
{
  "scenes": [
    {
      "id": "forest-morning",
      "name_zh": "森林早晨",
      "emoji": "🌲",
      "description": "清晨的森林，小鸟最先醒来唱歌",
      "sound_ids": ["animals.bird", "nature.stream", "nature.wind", "animals.woodpecker"],
      "ai_composed": true,
      "ai": { "model": "MiniMax-M3", "generated_at": "2026-06-29T12:00:00Z" }
    }
  ]
}
```

场景编排 Prompt：

```text
请从以下声音列表中挑选 3-6 个，组合成一个适合 1-6 岁儿童聆听的场景「{场景主题}」。
要求：
- 场景有叙事感，声音按时间或空间顺序排列；
- 只使用列表中存在的 sound_id；
- 返回 JSON 数组，元素为 sound_id。

可用声音列表：
{manifest.sounds 的 id + name_zh + tags}
```

场景会随 `manifest.json` 一起被前端读取，首页增加「🎬 场景」入口，点击进入后按顺序连续播放，每个声音高亮对应卡片。

#### 4.2.5 多音频随机轮播（播放体验增强）

针对拥有多条真实音频的动物条目，每次点击卡片或切换详情时，从 `sounds` 数组中随机抽取一条播放，避免孩子反复听到同一段声音，提升探索新鲜感。

**实现要点：**

- 前端维护一个当前声音的播放索引，每次触发播放时生成 `Math.floor(Math.random() * sounds.length)`。
- 若用户短时间内重复点击同一卡片，仍可随机到不同音频；连续两次相同不强制去重（简单可靠）。
- 随机逻辑纯前端实现，不依赖 AI，但音频列表来自 AI 生成/人工维护的 `meta.json`。
- 与 AI TTS 互不干扰：真实动物声音随机播放，TTS 朗读固定对应文案。

### 4.3 Phase C：AI 贡献辅助（详细实现）

#### 4.3.1 贡献者上传表单（`contribute.html`）

在 GitHub Pages 上新增一个纯静态贡献页面，流程：

1. 用户上传音频文件（本地预览）。
2. 填写声音名称（可选）。
3. 点击「让 AI 帮我填信息」。
4. 通过本地/云端代理调用 AI：
   - 根据文件名和音频特征（时长、文件名关键词）建议分类、子分类；
   - 推荐 emoji；
   - 生成 30 字描述和 30-50 字 `fun_fact`；
   - 生成标签建议。
5. 用户在表单中确认/修改后，生成一段可直接复制到 GitHub Issue 的 Markdown。

**AI 建议 Prompt：**

```text
用户上传了一段名为 "{filename}" 的音频，请为其建议声音条目的元信息。
要求：
- 分类必须从 {categories.json} 中选择；
- emoji 必须与内容相关；
- 描述不超过 30 字；
- 科普文案 30-50 字，适合 1-6 岁儿童；
- 返回 JSON 格式：{ category, subcategory, emoji, description, fun_fact, tags }
```

由于涉及 API Key，线上版本采用「预生成建议模板 + 用户提交后维护者本地运行 AI 辅助脚本」的混合模式。

#### 4.3.2 AI 音频质量初筛

在 PR 校验流程 `.github/workflows/validate-pr.yml` 中增加可选 AI 检查（不阻断合并）：

- 判断音频是否为真实录音（非合成音乐、非静音）。
- 检测明显底噪或爆音。
- 估算响度是否接近 -16 LUFS。
- 输出「建议通过 / 建议人工复核」标签。

实现方式：使用轻量级音频分析库（如 `ffmpeg` + `node`）做基础判断，复杂情况可调用 AI 辅助分析。

### 4.4 Phase E：AI 互动增强（详细实现）

#### 4.4.1 AI 猜声音游戏

**入口：** 首页顶部增加「🎮 猜声音」按钮。

**玩法：**
1. 系统从 `manifest.sounds` 中随机选 1 个答案。
2. 播放该声音的真实音频（从其 `sounds` 中随机一条）。
3. 展示 4 个候选卡片（emoji + 中文名），1 个正确答案 + 3 个干扰项。
4. 干扰项选择策略：优先同分类/同子分类/同标签，其次随机。
5. 孩子选择后：
   - 答对：撒花动画 + 随机鼓励文案（如「太棒了！这是小狗，它的鼻子可灵啦！」）。
   - 答错：高亮正确答案 + 播放正确答案音频 + 显示其 `fun_fact` + 温和提示「没关系，再听一次就记住啦」。

**AI 使用点：**
- 鼓励/提示文案可预生成模板库（20-30 条），也可调用 LLM 动态生成。
- 干扰项选择未来可升级为 LLM 语义推荐，但首期用规则即可。

**进度与勋章：** 答对计入「猜声音成就」，与现有探索进度系统打通。

#### 4.4.2 智能探索推荐

**入口：** 底部科普面板增加「✨ 你还可能喜欢」区域。

**实现：**
- 基于当前声音的标签、子分类、分类，计算相似度最高的 3 个声音。
- 相似度算法：共同标签权重 50% + 同子分类权重 30% + 同主分类权重 20%。
- 推荐结果以卡片形式展示，点击直接跳转并播放。
- 可预生成推荐语（如「小狗和狼都是犬科动物，听听狼怎么叫吧」），写入 `data/recommendations.json` 或前端通过模板生成。

#### 4.4.3 AI 声音伙伴（儿童问答）

**入口：** 底部面板或悬浮按钮「💬 问小精灵」。

**实现：**
- 预生成 20-30 个高频问题的 QA 映射，写入 `data/qa-cache.json`。
- 前端通过关键词匹配用户输入，命中则展示答案和相关声音卡片。
- 未命中时提示「这个问题好有趣，去问问爸爸妈妈吧」。
- 未来可扩展为本地代理调用 LLM 动态回答，但首期保持静态部署友好。

#### 4.4.4 AI 声音故事

**入口：** 首页「📖 听故事」模式。

**流程：**
1. 孩子选择 3-5 个声音（或选择系统推荐主题，如「农场的一天」）。
2. 调用 LLM 生成一段连贯的儿童故事，每个声音对应一个情节。
3. 播放时按故事顺序自动播放声音，并高亮对应卡片。
4. 故事文本以纯文字展示在底部面板；同时提供「🔊 AI 读故事」按钮，由 AI 朗读故事文字（不朗读真实动物声音）。

**故事 Prompt：**

```text
请用以下声音为 1-6 岁儿童编写一段 100-150 字的小故事。
要求：
- 每个声音在故事中出现一次，自然衔接；
- 语言口语化、有画面感；
- 不使用复杂成语；
- 只返回故事正文。

声音列表：
{sound_id + name_zh}
```

故事生成结果可缓存到 `data/stories/` 或 `data/scenes.json` 中，避免重复调用 LLM。

│   ├── ai-generate-scenes.js    # 场景编排

#### 4.5.2 新增本地/开发接口（可选）

| 端点 | 用途 | 说明 |
|------|------|------|
| `POST /api/ai/generate-fun-fact` | 为指定声音生成科普文案 | 仅本地开发 |
| `POST /api/ai/generate-scenes` | 生成场景编排 | 仅本地开发 |
| `POST /api/ai/suggest-meta` | 为上传音频建议元信息 | 仅本地开发 |
| `POST /api/ai/story` | 生成声音故事 | 仅本地开发 |
| `POST /api/ai/recommend` | 返回相似声音推荐 | 可前端规则实现，也可本地代理 |

> 线上 GitHub Pages 版本不依赖这些动态接口，只读取已缓存的 JSON 数据。

#### 4.5.3 缓存与成本控制

- **文本缓存**：`meta.json` 中 `ai.text_hash` + `ai.generated_at`，内容未变不重新生成。
- **场景/故事缓存**：生成后写入 `data/scenes.json` 或 `data/stories/`，随 `build-manifest.js` 一起构建。
- **Actions 触发**：仅在 `data/sounds/**/meta.json`、`.github/ai-config.json` 或 `data/categories.json` 变更时触发 AI 生成。
- **失败回退**：生成失败保留旧内容，并输出错误日志供人工排查。
